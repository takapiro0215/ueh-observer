Trust & Accuracy Contract v0.1

for UEH Observer Infrastructure

0. 目的（Purpose）

本システムは
実行・最適化・助言を行わない観測インフラである。

したがって精度の定義は
「予測の正しさ」ではなく、

観測状態の正確性・鮮度・整合性の保証

に基づく。

本契約は、
表示される全ての数値が
「どの程度信頼可能か」を明示するための基盤仕様である。

1. 精度の正式定義（Accuracy Definition）

本インフラにおける「精度」は以下の4要素で構成される。

1.1 Correctness（正確性）

  • プロトコルのオンチェーン状態と一致した値であること

  • 推定値ではなく、取得値を優先する

対象：

  • Health Factor

  • Collateral

  • Debt

  • Liquidation Threshold

1.2 Freshness（鮮度）

表示データは以下を必ず紐付ける：

  • 観測ブロック番号

  • 最終更新時刻

  • データ遅延（Lag）

例：

Block: 19283746

Lag: 0 blocks

Last Update: 14s ago

鮮度は「高速掲示板の信頼性」の核心とする。

1.3 Consistency（整合性）

単一ソースの値を盲信しない。
最低2系統の観測経路で整合性を検証する。

推奨：

  • Primary RPC

  • Secondary RPC / Indexer（将来）

整合条件：

  • ブロック差が閾値以内

  • 主要指標の乖離が許容範囲内

1.4 Provenance（出所透明性）

すべての表示はデータ出所を明示可能であること。

最低表示項目：

  • Chain

  • Protocol

  • Data Source（RPC/Provider）

2. 信頼メタデータ表示仕様（Board Metadata）

電光掲示板型CLIにおいて、
以下のメタ情報は必須表示とする。

必須メタ項目（MVP）

  • Data Block

  • Lag（blocks or seconds）

  • Source Status（OK / Degraded / Refusal）

例（標準表示）：

HF: 1.21

LiqDist: 8.3%

Block: 19283746 (lag 0)

Source: primary(ok) / secondary(ok)

Status: Stable

3. Refusal Protocol（拒否プロトコル）※UEH核心

3.1 Refusalの原則

不確実な状態での断定表示は禁止する。
信頼性が保証できない場合、
システムは「表示を拒否」または「劣化表示」に移行する。

これは障害ではなく、
信頼保護機構である。

3.2 Refusal発動条件（MVP基準）

A. データ整合性異常

  • Primary/Secondaryのブロック差 > 設定閾値

  • 指標乖離 > 許容範囲

表示：

Status: REFUSAL (inconsistent data sources)

B. RPC不安定 / タイムアウト

  • 応答遅延 > 設定秒数

  • 連続取得失敗

表示：

Status: DEGRADED (data source unstable)

C. 境界付近での不確実性

（例：HFが清算境界近傍）

条件例：

  • HF < 1.30 かつデータ遅延あり

表示：

Status: Boundary (verification strict mode)

4. Boundary-Aware Accuracy（境界適応型精度）

4.1 通常状態（Normal Zone）

  • 標準更新間隔（例：15–30秒）

  • 軽量検証

4.2 境界接近状態（Boundary Zone）

条件例：

  • Health Factor < 1.3

挙動：

  • 更新頻度向上

  • 二重検証強化

  • Refusal閾値を厳格化

これは「高速道路の事故掲示板」に相当する緊急精度モードである。

5. 更新モデル（Board Update Model）

本システムは
定周期観測型（Periodic Observer Model） を採用する。

初期推奨：

  • 15秒更新（MVP）

  • 非リアルタイム誤認を避けるため
    「Last Update」を明示表示する

理由：
過剰リアルタイム表示は
誤精度のリスクを増大させるため。

6. 非助言保証（Non-Advisory Integrity Clause）

本インフラは以下を絶対に行わない：

  • 投資助言

  • 最適行動提案

  • 自動実行

  • リスクスコアによる行動誘導

表示は常に：

状態・境界・観測値のみ

とする。

7. エラーハンドリング哲学（Survivability）

優先順位：

  1. 正確な表示

  2. 劣化表示（Degraded）

  3. Refusal（表示拒否）

  4. 沈黙（最終手段）

誤った断定表示は
最も重大な信頼侵害と定義する。

8. MVP適用スコープ（初期固定）

  • Chain: Ethereum

  • Protocol: Aave v3

  • Mode: CLI Board (Live Observer)

  • Access: Read-only / Non-custodial

  • Input: Wallet Addressのみ

9. 設計思想との整合宣言（UEH Alignment）

本契約は以下の思想と一致する：

  • Boundary（境界可視化）

  • Refusal（拒否による安全性）

  • Survivability（誤動作回避）

  • Observer（非介入観測）

したがって本システムは：

「便利ツール」ではなく
「信頼可能な観測インフラ層」である。
